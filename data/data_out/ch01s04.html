Принятие Python 3 на момент написания этой книги

Итак, хорошо ли благодаря новым интересным функциям Python 3 принят сообществом? К сожалению, пока нет. Популярная страница Python 3 Wall of Superpowers ( https://python3wos.appspot.com ), отслеживающая совместимость наиболее популярных пакетов с веткой Python 3, еще не так давно называлась Python 3 Wall of Shame. Эта ситуация меняется, и с каждым месяцем таблица перечисленных пакетов на указанной странице постепенно становится «более зеленой». Тем не менее, это не означает, что все команды, создающие свои приложения, в ближайшее время будут использовать только Python 3. Когда все популярные пакеты будут доступны на Python 3, популярное оправдание - пакеты, которые мы используем, еще не перенесены, - перестанет действовать.

Основная причина такой ситуации заключается в том, что перенос существующего приложения с Python 2 на Python 3 всегда является проблемой. Существуют такие инструменты, как 2to3, которые могут выполнять автоматическую трансляцию кода, но они не гарантируют, что результат будет на 100% правильным. Кроме того, такой переведенный код может не работать так же хорошо, как в исходной форме, без ручной настройки. Перенос существующих сложных баз кода на Python 3 может потребовать огромных усилий и затрат, которые некоторые организации могут себе позволить. Однако такие затраты можно разделить по времени. Некоторые хорошие методологии проектирования архитектуры программного обеспечения, такие как сервис-ориентированная архитектура или микросервисы, могут помочь в достижении этой цели постепенно.

В конечном итоге переход на Python 3 может оказать только положительное влияние на проект. Согласно PEP-404, в ветке Python 2.x больше не будет версии 2.8. Кроме того, в будущем может наступить время, когда все основные проекты, такие как Django, Flask и numpy, потеряют совместимость с 2.x и будут доступны только на Python 3.

Мое личное мнение на эту тему можно считать противоречивым. Я думаю, что лучшим стимулом для сообщества было бы полное прекращение поддержки Python 2 при создании новых пакетов. Это, конечно, сильно ограничивает возможности такого программного обеспечения, но это может быть единственный способ изменить образ мышления тех, кто настаивает на использовании Python 2.x.
